## 스크립트

에이전트에게 여러 출처에서 제공되는 다양한 도구를 제공하는 것은 흔한 일이며, 에이전트는 주어진 작업을 해결하기 위해 필요에 따라 도구를 사용합니다 . Strands에서는 에이전트에게 도구 접근 권한을 부여하는 여러 가지 방법이 있습니다. 이미 내장된 벤트 도구를 사용하는 방법과 툴 데코레이터를 사용하여 파이썬 함수를 도구로 바꾸는 방법을 살펴보았습니다 . 또 다른 패턴은 MCP입니다. MCP( Model Context Protocol)는 에이전트가 외부 기능을 일관된 방식으로 검색하고 상호 작용할 수 있도록 하는 개방형 표준입니다 . 에이전트가 GitHub, AWS, API, 브라우저 도구 또는 회사 내부 시스템과 상호 작용하도록 만들고 싶을 수도 있습니다. 각 기능을 도구별로 수동으로 연결하는 대신 , 표준 프로토콜을 사용하여 해당 기능을 이미 제공하는 MCP 서버에 연결합니다. 최신 에이전트 시스템은 모든 것을 프롬프트 나 로컬 도구에 직접 인코딩하려고 하기보다는 런타임에 외부 지식과 기능에 액세스하는 데 점점 더 의존하고 있습니다. 자, 이제 AWS 기술을 활용한 심도 있는 기능을 갖춘 간단한 코딩 도우미 에이전트의 예시를 살펴보겠습니다. AWS는 AWS 에이전트 툴킷의 일부로 MCP 서버를 제공하여 AWS 기능을 에이전트에 직접 노출합니다. AWS 관련 지식을 프롬프트에 직접 입력하거나 자체 검색 시스템을 유지하는 대신, 에이전트는 MCP 도구를 통해 필요한 정보를 동적으로 가져올 수 있습니다. 연결 방법을 보여드리겠습니다. Strands는 원격 또는 로컬 MCP 서버에 연결할 수 있는 MCP 클라이언트를 제공합니다 . MCP 패키지에서 스트리밍 가능한 HTTP 클라이언트를 가져오고, Strands에서 MCP 클라이언트를 가져옵니다. 다음으로 MCP 클라이언트를 생성하고 AWS MCP 서버 엔드포인트를 가리키도록 설정합니다. 클라이언트 정보를 확보하면, 이처럼 에이전트의 도구 목록에 추가합니다. 에이전트가 초기화될 때, Strands는 MCP 서버에서 제공하는 도구를 자동으로 검색하여 모델링에 사용할 수 있도록 합니다. 그러면 "인증 및 파일 업로드 기능을 갖춘 서버리스 고속 API 백엔드를 구축해야 합니다 . 어떤 AWS 서비스를 사용해야 하며, 어떻게 아키텍처를 설계해야 할까요?"와 같은 질문을 할 수 있습니다. 에이전트는 로컬, 원격, 타사 호스팅 또는 자체 개발 서버 등 다양한 MCP 서버에 동시에 연결할 수 있습니다. 하지만 시스템이 복잡해질수록 도구와 MCP가 많은 컨텍스트와 토큰을 소비할 수 있다는 것을 알게 될 것입니다. 도구 설명, 스키마 및 MCP 서버 메타데이터는 모두 로드 될 때 모델의 컨텍스트 창에서 공간을 차지합니다 . 문제는 에이전트에 너무 많은 도구를 과부하시키면 도구 선택 판단력이 떨어지거나 심지어 존재하지 않는 도구 이름을 만들어내는 오류를 범할 수 있다는 것입니다 . 훌륭한 하네스 디자인의 핵심은 모델에게 과도한 부담을 주지 않으면서 필요한 기능들을 제대로 발휘할 수 있도록 하는 것입니다. 이것이 바로 도구 필터링이 유용해지는 이유 중 하나입니다. 예를 들어, 일부 MCP 서버는 수십 가지 도구를 제공하지만 에이전트에는 그중 몇 가지만 필요합니다. 도구 필터링을 사용하면 상담원에게 전달할 관련 도구만 선택할 수 있으므로 컨텍스트 정보의 과다함을 줄이고 혼란을 방지할 뿐만 아니라 비용도 절감할 수 있습니다. 다음은 동일한 AWS MCP 서버를 사용하고 모든 도구를 출력하는 스크립트입니다. 그런 다음 동일한 MCP 서버를 사용하여 Strands의 도구 필터를 이용해 목록을 필터링합니다. 허용 목록, 거부 목록 또는 정규 표현식 패턴을 전달할 수 있습니다. 그럼 실행해 봅시다 . 에이전트가 실행되었고 MCP 서버에 11개의 도구가 나열된 것을 확인할 수 있지만, 필터링 후에는 2개의 도구만 남겨 에이전트에 제공했습니다 . 그래서 우리는 그 도구 목록을 줄일 수 있었습니다. 어떤 에이전트에게 어떤 작업을 수행할지 결정하는 것은 바로 여러분이 만든 하네스에 내장할 수 있는 기능 중 하나입니다. 더욱 고급 시스템에서는 워크플로 및 MCP 서버 유형에 따라 의미론적 또는 동적 도구 필터링을 구축할 수도 있습니다 . 이 예제에서는 AWS에서 호스팅하는 MCP 서버를 사용하지만, 동일한 방식으로 자체 MCP 서버를 구축하고 Strands에 연결할 수도 있습니다 . 자, 여기까지가 MCP 관련 도구들입니다. 다음 영상으로 넘어가기 전에, 도구에 대해 여러분이 이해해주셨으면 하는 점이 있습니다. 내장 도구, 커뮤니티에서 만든 도구 또는 직접 도구를 만들 때, 도구가 호스트 프로세스의 권한으로 실행된다는 점을 이해하는 것이 좋습니다. 즉, 에이전트에게 셸 도구에 대한 무제한 액세스 권한을 부여하면 사실상 시스템 전체에 대한 액세스 권한을 부여하는 것과 같습니다 . curl 명령어를 실행할 수 있는 에이전트는 클라우드 자격 증명을 읽고, 내부 네트워크에 접근하고, 의도치 않게 노출시킨 파일을 덮어쓸 수도 있습니다. Strands에는 이러한 보안 문제를 해결하는 데 도움이 되는 Strands Shell이라는 기능이 있습니다. Strands Shell을 사용하면 셸 프로세스가 사용자가 제어하는 ​​격리된 샌드박스 내에서만 실행되도록 제한할 수 있습니다. 에이전트가 접근할 수 있는 파일, URL, 자격 증명 등 필요한 모든 것을 사전에 선언하면, 에이전트 입장에서는 그 외의 모든 것은 존재하지 않는 것처럼 보입니다 . 그리고 MCP와 관련해서는 보안 문제가 있습니다. MCP 도구를 에이전트에 연결할 때, 이러한 도구는 다른 사람이 제어하는 ​​도구일 가능성이 있습니다. 이는 도구 로직을 바꾸거나 사용자가 에이전트에 제공할 의도가 없었던 새로운 도구를 추가할 수 있음을 의미합니다 . 실제 운영 시스템에서 도구 접근 권한은 보안 경계의 일부가 됩니다 . 그러므로 필요할 때 Strands Shell을 사용하고 , 에이전트에 연결하는 MCP 서버와 연결 방식에도 주의를 기울이십시오. 도구에 대한 사고 모델은 다음과 같습니다. 이 하네스를 사용하면 에이전트가 프로젝트별 로직 및 통합을 위한 사용자 지정 도구 , 일반적인 기능을 위한 커뮤니티 또는 공급업체에서 제공하는 내장 도구, 그리고 외부 시스템에 연결하기 위한 MCP 서버에 액세스할 수 있습니다 . 이 모든 기능은 동일한 에이전트의 도구 목록 내에서 공존할 수 있습니다 . 좋아요. 그래서 이 시점에서 우리는 추론하고, 도구를 사용하고, 외부 서비스에 연결할 수 있는 에이전트를 갖게 되었고, 그 뒤에 있는 모델을 교체할 수도 있습니다. 하지만 우리는 그것이 해당 도구들을 어떻게 사용하는지에 대해서는 통제할 수 없습니다. 파일을 삭제 하거나 이메일을 보내기로 결정하면, 그냥 그대로 실행합니다. 에이전트 루프를 가로채고, 무슨 일이 일어날지 검사하고 , 심지어 파괴적인 작업이 실행되기 전에 담당자에게 문의할 수 있도록 하는 훅을 사용하여 이 문제를 해결할 것입니다.
