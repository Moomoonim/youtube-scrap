## 스크립트

When is the added complexity of multi vector search actually worth it? Multiv vector representations require more storage, more computation and more careful implementation than simple single vector embeddings. So why bother? The answer comes down to one core capability fine grain matching. In the previous lessons, we learned how late interaction preserves token level representations and how Maxim computes similarity through independent token matching. Now, let's see when this precision actually matters and when it doesn't. Single vector embeddings capture topical similarity. It's like summarizing a book in one sentence. You get a general theme but lose the nuances that might be relevant to a specific query. This compression is a form of lossy averaging. Multiv vector search does token level verification. Instead of averaging everything together, it maintains separate representations for each concept. Every requirement in your query needs to find a strong match for the overall score to be high. This matter most for multi-requirement queries. Consider Python async database connection pooling. This has multiple specific concepts that all need to be present. Single vector search might find documents that are topically related, but multi vector search verifies that each concept is actually addressed. When multivactor search wins for complex queries with multiple specific requirements, not just what is this about, but does it address all the things when your the queries need token level verification, not just topical matching. Distinguishing comprehensive answers from keyword stuff content. A document mentioning the right words superficially won't rank as high as one that actually addresses them. Max seems requirement that every query token finds a strong match prevents keyword staffing from inflating scores. When contextualized precision matters. When bank needs to mean the right kind of bank for your query. when the same word should match differently depending on context. You may also wonder when to use single vector. Definitely for simple broad queries, it should work fine with single vectors. Queries like Python tutorials or machine learning basics are seeking documents about a topic not verifying specific requirements. The added precision of token level matching doesn't provide significant value. Also in resource constraint environments, multi vector search has significant storage and compute overhead, you need to store multiple vectors per document and maxim requires computing similarities across all query document token pairs. Sometimes topical similarity is sufficient. If you just need documents about a topic rather than documents that address specific requirements, single vector search is faster, cheaper, and usually good enough. Multiv vector search excels at fine grain matching scenarios where every query token needs to find a strong match in the document. The keyword stuffing problem that pla single vector search is naturally addressed by token level verification. But we've hinted at significant costs. In the next lesson, we'll examine the problems of multi vector search head-on. The memory explosion from storing multiple vectors per document. the computational overhead of Maxim and the fundamental indexing challenge that forces brute force scans. This isn't meant to discourage you. It's meant to prepare you.
