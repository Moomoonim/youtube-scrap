## 스크립트

We've built an agent with tools, but when the&nbsp; agent decides to call a tool, it just calls it.&nbsp;&nbsp; There's nothing you can do about it.&nbsp; Models are getting smarter all the time,&nbsp;&nbsp; and they're very capable. But capability isn't&nbsp; the same as reliability. You can't guarantee&nbsp;&nbsp; the model won't do something destructive,&nbsp; call the same tool in an infinite loop,&nbsp;&nbsp; or pass bad inputs to a sensitive API without&nbsp; guardrails in place. Designing an agent harness&nbsp;&nbsp; thoughtfully allows you to tackle these problems.&nbsp; And one of the main ways to mitigate these risks&nbsp;&nbsp; is to inject deterministic code into the agent&nbsp; life cycle at specific points through hooks.&nbsp;&nbsp; They give you a place where you can enforce&nbsp; the things that can't be left to probability&nbsp;&nbsp; like approval gates, rate limits, input&nbsp; validation, audit logging, and safety checks.&nbsp;&nbsp; Hooks are like the middleware for the agent&nbsp; life cycle. They exist for the same reason&nbsp;&nbsp; web frameworks have middleware. You don't just&nbsp; trust every request handler to check off or log&nbsp;&nbsp; correctly. So you put that in a middleware layer&nbsp; that it's guaranteed and it's applied reliably.&nbsp;&nbsp; Hooks are that layer but for agents. You've&nbsp; actually already seen hooks in action in the&nbsp;&nbsp; first video when the coding agent paused and asked&nbsp; for approval before running certain operations.&nbsp;&nbsp; That approval workflow was implemented&nbsp; through hooks. The way hooks work is&nbsp;&nbsp; pretty straightforward. You write a callback&nbsp; function, register it for a life cycle event,&nbsp;&nbsp; and your function gets called whenever the event&nbsp; fires. Multiple hooks can listen to the same&nbsp;&nbsp; event, so you can stack behaviors. You can have&nbsp; one hook for logging, another for validation,&nbsp;&nbsp; and another for approval gates without&nbsp; them stepping on each other or interfering.&nbsp;&nbsp; Human in the loop is one of the most common&nbsp; reasons to use hooks. Let's say you're building an&nbsp;&nbsp; agent that runs locally on someone's machine. And&nbsp; one of the things it can do is manipulate files.&nbsp;&nbsp; What you might want is a hook that pauses the&nbsp; agent before any file deletion happens and asks&nbsp;&nbsp; the user for approval. The user can see what's&nbsp; about to be deleted, approve or deny the action,&nbsp;&nbsp; and the agent either continues or cancels based on&nbsp; that response. Let's look at some code to do this.&nbsp;&nbsp; For the agent itself, we have several tools for&nbsp; file management including a delete file tool.&nbsp;&nbsp; Then we have a hook defined here. This is a hook&nbsp; called delete approval hook which subclasses hook&nbsp;&nbsp; provider from strands. Hook provider is a base&nbsp; class that gives you a structured way to register&nbsp;&nbsp; hook callbacks. Inside register hooks, we wire&nbsp; up our callback to the specific life cycle event&nbsp;&nbsp; we care about. In this case, we register check&nbsp; delete to fire on every before tool call event.&nbsp;&nbsp; Inside check delete, the first thing&nbsp; we do is inspect the tool name.&nbsp;&nbsp; If the tool isn't delete file, we immediately&nbsp; return and allow execution to continue normally.&nbsp;&nbsp; But if the tool is attempting a delete&nbsp; operation, we call event interrupt.&nbsp;&nbsp; This pauses the agent loop and returns control&nbsp; back to our application. Your application can&nbsp;&nbsp; then prompt the user for approval before resuming&nbsp; with the agent operations. If the user declines&nbsp;&nbsp; the action, we call event.canc tool with a message&nbsp; explaining why the tool execution was blocked.&nbsp;&nbsp; That cancellation message gets passed back to the&nbsp; agent loop as a tool result and the model adapts&nbsp;&nbsp; its behavior accordingly. Then during agent&nbsp; initiation, we pass the hook into the agent&nbsp;&nbsp; using hooks equals delete approval hook and the&nbsp; agent harness handles the interrupt life cycle,&nbsp;&nbsp; checking if the agent paused, prompting the user&nbsp; and feeding the response back to the agent until&nbsp;&nbsp; the request completes. Let's run it. First, I'll&nbsp; ask the agent to create a file called data.txt.&nbsp;&nbsp; The agent uses the file tools normally. Then&nbsp; I'll ask the agent to delete the data.txt&nbsp;&nbsp; file. A hook fires before the delete tool&nbsp; executes and pauses the loop. The application&nbsp;&nbsp; asks for approval and I'll type n to deny&nbsp; the operation. The hook cancels a tool call,&nbsp;&nbsp; passes the cancellation back into the loop, and&nbsp; the agent continues without deleting the file.&nbsp;&nbsp; Now let's look at another kind of hook using the&nbsp; same overall structure. We subclass hook provider&nbsp;&nbsp; register a before tool call event and this time&nbsp; instead of interrupting for human approval we&nbsp;&nbsp; track how many times each tool has been called&nbsp; during the request. The limit tool counts hook&nbsp;&nbsp; accepts a max calls threshold and stores counts&nbsp; for each tool. On every tool call, it increments&nbsp;&nbsp; the count and checks whether the tool exceeded&nbsp; the allowed limit. If the limit is exceeded,&nbsp;&nbsp; it calls event.canc cancel tool with a message&nbsp; instructing the model to stop calling that tool.&nbsp;&nbsp; The model sees that cancellation as the tool&nbsp; result and adjusts its behavior accordingly.&nbsp;&nbsp; This is useful because agents can occasionally get&nbsp; stuck in loops repeatedly calling the same failing&nbsp;&nbsp; tool. A lightweight deterministic safeguard&nbsp; like this can prevent runaway costs and unstable&nbsp;&nbsp; behavior. This is one of the main reasons why&nbsp; tuning your harness is important for cost control&nbsp;&nbsp; and reliability. These are simple examples, but&nbsp; hooks are one of the foundational mechanisms&nbsp;&nbsp; that make production agents safe, observable,&nbsp; and controllable. They're also the foundation&nbsp;&nbsp; for plugins, which package hooks, tools, and&nbsp; context management into reusable components,&nbsp;&nbsp; which we'll be looking at next. Starting in&nbsp; the next video, we'll begin building a customer&nbsp;&nbsp; service agent to show that these same patterns&nbsp; and ideas apply across different use cases.&nbsp;&nbsp; That agent will become one of the main&nbsp; running examples for the rest of the course.
