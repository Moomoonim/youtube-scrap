## 스크립트

You've learned how Maxim enables fine grain matching and explored both the benefits and challenges of multivactor search. Now it's time to put that knowledge into practice. Quadrant provides first class support for multi vector embeddings. In this lesson, you'll learn how to configure collections for multi vector search index documents with token level embeddings and execute queries using max C. By the end, you will understand the key configuration parameters, storage optimization options, and be ready to build your own multi vector search applications. Already, this is how we set up a collection to support multi vector search. The size parameter is 228. This is the dimensionality of each individual token embedding. And Colbert uses 228 dimensional vectors for each token. This is more the typical single vector models because we are storing many vectors per document. The distance is set to dot for dot product. This is the similarity function used for comparing individual tokens. When maxim computes similarity between a query token and the document token, it will use this function. The multi vector config with maxim tells quadrant that this isn't a regular single vector index. We're using multi vector mode with the maxim comparator. This tells Quadrant to use the maximum distance metric we covered earlier. The HNSW config with M set to zero disables HNSW indexing. Remember from our previous lesson, HNSW doesn't work with maximing because of the asymmetric problem and query dependency. Setting M to zero tells Quadrant not to build an index, saving memory and indexing time. When you ingest the data, you can use models document and quadrant will handle embedding generation via fast embed. You don't need to manage embedding generation in your application code itself. You just provide a text and model name, one of the supported ones. Quadrant tokenizes and embeds client side. The payload field should obviously store the original text and animated data you want to associate with the document. Because of the collection configuration, our query results will be ranked by maximum score. Higher scores mean better matches across all query tokens. Behind the scenes, Quadrron computes maxim exactly as we discussed. So for each query token, it finds the best match among document tokens and sums those maximum similarities. The simplest optimization possible is obviously to offload the data to disk. Setting on disk to true, store vectors on disk instead of RAM. It's essential for large collections where keeping everything in memory isn't feasible. Instead of needing hundreds of gigabytes of RAM, you can use disk storage. The trade-off is lower memory usage but higher latency. Reading vectors from disk is slower than reading from RAM. And query times increase from milliseconds to potentially seconds depending collection size and disk performance. For production systems that need subsecond response times, disk offloading alone isn't as usually sufficient. Combine disk storage with module 3 optimization techniques like quantization pooling, move vera, and multisetch retrieval.
