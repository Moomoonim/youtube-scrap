Most AI applications fail. Not because the model isn't good enough and not because the prompts are poorly written. They fail because of how information flows through the system or more accurately how it doesn't. That problem has a name and it's called context engineering. And if you're building anything serious with AI right now, it's probably the most important thing you need to know. Agents and LLMs have one fundamental constraint, the context window. This is the model's active working memory, the finite [music] space where it holds everything it needs to do its job. The instructions, the conversation history, [music] the retrieved documents, the tool outputs, all have to live within this block of text [music] and every token you put in there costs up space. And when that space fills up, something is dropped or missed or glazed over. The [music] instinct here is to assume that this gets solved by bigger context windows. Unfortunately, it doesn't. Longer contexts actually introduce [music] their own failure modes. When you get closer to larger context hitting those upper limits, you'll [music] start seeing higher hallucination rates. The model will over rely on early context [music] and ignore later inputs and contradictory information will start pulling the model's reasoning [music] in different directions. So, 1 million token context window really doesn't give you a smarter [music] agent. It's just more that the agent can get wrong. So, if you can't scale the context window, what can you do? You can build a system around the model that controls what information reaches the model, when it reaches it, and in what form. This is context [music] engineering. Context engineering has five main layers, each solving a different problem within the pipeline. The first is query augmentation, [music] the first step in any generation pipeline. When a user sends a message, that message is almost never in the right format for the rest of your pipeline. Users tend to be very vague. [clears throat] They use shorthand, they ask multipart questions without realizing it. They don't use proper grammar, they misspell words, or a million other things. So, query augmentation is a step where we translate the messy human input into something the system can actually use [music] well. This can mean rewriting the query into more clear terms so that the retrieval system [music] can actually match it against relevant documents. It can mean expanding it, so generating related terms and phrasing things that capture what the user probably meant even if they didn't say it exactly. Or it can mean decomposing it, so breaking a complex multi-part question into a set of focused sub-questions that can each be answered independently and then synthesized together. But, one of the most powerful and foolproof ways to handle this is through a query [music] agent. Rather than applying a fixed set of augmentation rules, a query agent uses an LLM to reason about the user's intent [music] and decide dynamically how to query your data. So, at Weaviate we built this into our query agent. It takes a natural language question from the [music] user. It understands the structure of your data collections and then decides which collections are relevant. [music] It'll construct the right query including filters and aggregations, and then evaluate whether the results actually answer the question in [music] the first place. And if they don't, it'll reformulate the queries and try again. It can also handle follow-up questions because it [music] keeps awareness of the conversation context. What that means in practice is that your users don't need to know how your data is structured so they [music] can query your database with the right words and filters. They simply ask in natural language and the agent will figure it all out for them. It handles the messy, incomplete, real-world queries in ways that fixed pipelines just simply can't. No amount of sophisticated retrieval fixes a misunderstood intent. If you put a bad query into your retrieval system, you won't get the right context back and your agentic system will fail. Garbage in, garbage out. And that brings us directly into our second layer, retrieval. So, LLMs are trained on a certain set of data, meaning they probably know next to nothing about your private documents, your company's knowledge base, and last week's incident reports. [music] Retrieval is how we bridge that gap, but there's a lot of nuance to do it well. [music] How you prepare your data for your database has one of the highest impacts on retrieval quality. &gt;&gt; [music] &gt;&gt; So, this is where chunking comes in. Chunking is a process of splitting your documents into pieces that can be embedded and retrieved individually. [music] The challenge is that you're balancing two competing requirements. Chunks need to be small enough to retrieve precisely, but large enough to carry meaningful context when they land in the model's window. The simplest chunking approach is fixed-size chunking with overlap. So, you split your documents every so many tokens with a sliding overlap between chunks so you don't lose any information. [music] This is a reasonable starting point, but it's completely blind to the document's actual structure. &gt;&gt; [music] &gt;&gt; So, it'll split up in the middle of a paragraph or separate a question from its answer or break information that should actually stay together. A step up from that is recursive chunking. [music] So, splitting first by paragraphs, then sentences, then words if needed, working down through the document's separators until the chunk fits in your target [music] size. This respects structure better than fixed-size without requiring you to know anything [music] specific about the document formats. Along the same lines, but a little bit more advanced, is semantic chunking. But, instead of structure, we split based on meaning. So, we embed windows of sentences [music] and measure how much the meaning shifts between those windows. When it detects a significant shift in the topic, it creates [music] a new chunk. The result is chunks that are semantically coherent. So, each one is about one thing, which tends to produce much cleaner retrieval. &gt;&gt; [music] &gt;&gt; This, however, is a lot more overhead in terms of preprocessing than recursive chunking or fixed-size. [music] The right chunking strategy depends on your documents and your use case. There's a lot of options, but the best you can do is start easy, get everything working, and then benchmark different strategies with a subset of your data when you're ready for an upgrade. The third layer of our context engineering system is memory. &gt;&gt; [music] &gt;&gt; An LLM with no memory treats every session as a blank slate. This works somewhat well for a demo. In production though, it means your agent can't learn from past interactions, it [music] can't maintain continuity across sessions, and it can't build any persistent understanding of its environment or its users. &gt;&gt; [music] &gt;&gt; Every conversation starts from zero. A lot of pipelines I've seen just dump the entire conversation history into the context window and call it memory. Sure, this may solve the continuity problem, but the window will also very quickly fill up, and [music] then we'll run into all the problems I mentioned in the beginning of the video. Lost context, model confusion, or bias towards the [music] beginning of the context. Beyond that, a long undifferentiated history is noisy. Older information that's no longer relevant is competing [music] for attention with the things that actually matter right now, and the model will just get confused. This is where the difference between short-term [music] and long-term memory comes in. Short-term memory is what lives in the context window. So, the active conversation, recent tool outputs, the immediate reasoning state. [music] Long-term memory is stored externally, so typically in something like a vector database, and retrieved only when [music] it's relevant. This is where we add information that we want persisted across sessions. So, user preferences, past interactions, domain knowledge, [music] etc. Memory isn't just about storage, it's about maintenance. &gt;&gt; [music] &gt;&gt; A memory store that you only ever write to becomes stale, contradictory, [music] and expensive to search over time. There are four ways we can solve this. First, pruning, so removing information that's outdated or no longer relevant. [music] Second, deduplication by merging entries that are saying the same things in slightly different ways. Third is resolving conflicts where [music] newer information contradicts older information. And fourth, and most important, is to deliberately forget things. [music] Because holding on to certain information that's not useful anymore will actually make your system worse. The goal is a memory system that doesn't just accumulate history, [music] but actively maintains an accurate, current, useful representation of what the agent knows. So, LLMs are disconnected from live data and live system, but tools are what gives them that reach. The ability to query a database, or call an API, write to a file, trigger workflow, interact with the world outside of the context window. The way this works in practice is through function calling. When a model decides it needs to use a tool, it generates a structured output, a function name and the argument to pass to it. Your system then executes that and feeds the results back into the context window. The model will see the results, decide whether it's sufficient, and either continues or try something else. That loop, thought, action, observation, is the fundamental reasoning cycle that makes agentic systems work. There are two approaches to tool integration that are worth distinguishing. MCP, the Model Context Protocol, is a standardization layer that lets any MCP-compatible application connect to any MCP server through a common interface. So, instead of building a custom integration for every external system, you write once to the standard. It's become pretty common practice to use MCP when building agent tool integrations. Agent skills is another standard for tool use. These are reusable, domain-specific capabilities, &gt;&gt; [music] &gt;&gt; things like the ability to search your vector database intelligently, or apply a specific domain's reasoning patterns. Compared to MCP, these are less structurally defined functions, and more a set of guides and design patterns that the agent can call into its context when it needs more information. Tools are actually where a lot of agent failures happen in production. The model will select the wrong tool because the descriptions were unclear or formulate the arguments incorrectly because the expected format wasn't explicit enough. It'll miss available tools entirely because too many were loaded into the context at once and the relevant one got lost. These are all context engineering failures. A well-written tool description, a well-managed tool registry, and a careful orchestration of which tools are visible at which points in the reasoning loop are the things that determine whether your agent uses its tools reliably or unpredictably. The fifth and final layer of context engineering that ties all of these things together [music] is so query augmentation, retrieval memory, and tools is agents. Agents are the decision-making layer, the component that determines how information flows through all of the above. They dynamically evaluate what you need to know, decide what they need, route to the right tools, and adjust when something goes wrong. [music] It's not a fixed system, and this is also part of why they're so hard to build. [music] So, when context management breaks down anywhere in the system, when the retrieval returns the wrong chunks, or when [music] memory surfaces stale information, when a tool is misused, the agent is the thing that compounds that failure further down the pipeline.
