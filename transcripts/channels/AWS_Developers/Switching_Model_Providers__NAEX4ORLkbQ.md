## 스크립트

모델마다 장점이 다릅니다. 일부는 코딩 작업이나 장시간 실행되는 워크플로에 최적화되어 있는 반면, 다른 일부는 계획 수립, 심층 연구에 더 적합 하거나 속도와 낮은 지연 시간 작업에 최적화되어 있습니다. 구축하는 에이전트 시스템이 더욱 정교해짐에 따라, 동일한 시스템 내에서 서로 다른 에이전트에 서로 다른 모델을 적용하는 것이 일반적입니다. 오케스트레이션에는 하나의 모델을 , 특수 코딩 작업에는 다른 모델을, 검증 또는 평가 단계에는 또 다른 모델을 사용할 수 있습니다. 그리고 어떤 경우에는 평가와 같은 서로 다른 작업에 완전히 다른 모델 제공업체를 사용하는 것이 유용할 수 있습니다. 이렇게 하면 생성과 검증 모두에 동일한 모델과 학습 패턴에 의존하지 않게 됩니다 . 이러한 다중 에이전트 시스템을 구축하기 위한 전략은 과정 후반부에 다루겠지만, 지금 중요한 것은 모델 선택이 아키텍처의 일부가 된다는 점을 이해하는 것입니다. 왜냐하면 모델마다 강점, 비용, 지연 시간 특성 및 도구 사용 방식이 다르기 때문입니다. 이것이 바로 Strands가 모델 제공자를 공통 인터페이스 뒤에 추상화하는 이유 중 하나입니다 . 모델 클래스를 인스턴스화하고 에이전트에 전달하면 나머지는 모두 동일하게 유지됩니다. 어떻게 작동하는지 살펴보겠습니다. 기본 제공업체인 Amazon Bedrock부터 시작하겠습니다. 모델 인수를 지정하지 않고 에이전트를 생성하면 Strands는 US West 2의 Claude Sonnet 4가 포함된 Bedrock 모델을 사용합니다. 실제 운영 환경에서는 일반적으로 사용할 모델 과 구성 방식을 명시적으로 지정하는 것이 좋습니다. 그래서 대신 Bedrock 모델을 가져온 다음, 우리 모델의 인스턴스를 생성하고 Claude Opus 4 6과 같은 특정 모델 ID를 전달한 다음, 에이전트 인스턴스를 생성할 때 해당 ID를 직접 전달합니다. 이제 한번 실행해 볼 수 있겠네요. 현재 에이전트는 기본 모델 대신 Opus를 사용하고 있지만, Bedrock에서 실행하는 대신 Anthropic의 API를 직접 사용하고 싶을 수도 있습니다 . 음, 패턴은 그대로 유지됩니다. 먼저 Anthropic 모델을 임포트한 다음 , Anthropic API 키를 전달하여 Anthropic 모델의 인스턴스를 생성할 수 있습니다. 그런 다음 이 인스턴스를 Bedrock 모델 대신 에이전트에 전달하면 됩니다 . 이것을 OpenAI 모델로 바꿔도 똑같이 작동합니다. 자, 이제 Ollama에서 로컬 모델을 사용하여 Strands로 설정할 수 있는 또 다른 흥미로운 옵션을 보여드리겠습니다 . Ollama를 사용하면 클라우드 추론 API 없이 자체 하드웨어에서 모델을 직접 실행할 수 있습니다. Strands에서는 이와 같이 Ollama 모델 공급자를 생성한 다음 실행 중인 로컬 Ollama 서버를 가리키도록 설정하면 됩니다. 이미 로컬에서 Gemma가 실행 중이므로, 이 모델 공급자를 에이전트에 제공하고 파일을 저장한 다음 실행하면 됩니다 . 로컬 모델은 저비용 실험, 오프라인 개발 및 개인 정보 보호에 민감한 워크플로에 유용합니다. 하지만 그에 따른 절충점도 있습니다. 로컬 모델은 일반적으로 프론티어에서 호스팅되는 모델보다 크기가 작고 기능이 떨어지며 , 도구 호출 품질은 모델에 따라 크게 다를 수 있습니다 . 자신의 특정 사용 사례에 맞춰 신중하게 테스트해야 합니다. 다음 영상에서는 에이전트가 외부 시스템 및 기능에 연결하는 주요 방법 중 하나인 도구 및 모델 컨텍스트 프로토콜( MCP)에 대해 자세히 살펴보겠습니다 .
