GitHub Actions를 사용하여 취약점 검사, 테스트 실행, 릴리스 생성, 팀 에 중요한 업데이트 알림 보내기 등 개발자 워크플로에서 흔히 수행하는 작업을 자동화하는 방법에 대해 이야기해 보겠습니다. GitHub UI에서 포크를 생성하는 방법과 포크를 최신 상태로 유지하는 방법은 다음과 같습니다. 복사하려는 저장소를 엽니다. 오른쪽 상단에서 포크를 선택하세요. 분기점을 어디에 만들지 선택하세요 . 원한다면 이름을 지어주거나 원래 이름을 그대로 사용해도 됩니다. 포크 생성 옵션을 선택하세요. 이제 귀하의 계정에서 프로젝트의 전체 사본을 확인하실 수 있습니다. 로컬에서 작업하려면 코드 버튼을 선택 하고 해당 코드를 컴퓨터로 복제하세요. gh.io/gfbforks에서 [음악]에 대해 더 자세히 알아보세요. GitHub UI에서 포크를 최신 상태로 유지하려면 다음 단계를 따르세요. GitHub에서 포크를 열어보세요. 파일 목록 위에서 동기화 포크 메뉴를 엽니다. 업데이트 분기를 선택하세요 . GitHub는 원본 저장소의 최신 변경 사항을 사용자의 포크에 가져옵니다. 이렇게 하면 사본이 최신 상태로 유지됩니다 . 사용자 인터페이스 대신 명령줄을 통해서도 이 작업을 수행할 수 있습니다 . 먼저 원본 프로젝트를 업스트림 원격 저장소로 추가합니다 . 다음 명령어를 실행하세요. 원격 저장소 추가 업스트림을 가져와서 원래 저장소 URL을 추가하세요. 그런 다음 최신 변경 사항을 가져오고 `get fetchupstream` 및 `get merge upstream/main`을 실행합니다. 또는 프로젝트에서 다른 기본 브랜치를 사용하는 경우 해당 브랜치를 병합합니다 . 이렇게 하면 원본 프로젝트의 최신 업데이트가 포크된 로컬 복사본에 적용됩니다. ` git push` 명령어를 사용하여 병합된 변경 사항을 GitHub 저장소에 푸시하면 포크한 저장소가 동기화됩니다. 글렌다, 깃허브에서 풀 리퀘스트를 어떻게 검토하나요? 풀 리퀘스트 (PR)는 코드를 공유하고 변경 사항에 대해 논의하는 곳입니다. PR을 검토할 때 도움이 되는 세 가지 방법을 소개합니다. 먼저, 홍보의 목표를 이해하는 것부터 시작하세요 . 풀 리퀘스트를 열고 설명을 읽어보세요. 관련된 문제를 찾아보세요. 작성자의 스크린샷이나 메모 . 목적을 알면 변화가 필요한 이유와 원하는 바를 이해하는 데 도움이 됩니다 . 다음으로, 코드를 작은 부분씩 나누어 검토하세요. 파일 변경 탭을 엽니다. 변화 과정을 그룹별로 차근차근 진행하세요. 이해가 안 되는 부분이 있으면 댓글로 남겨주세요 . 질문을 하거나, 제안을 하거나, 더 나은 접근 방식이 있다면 저자에게 알려주세요 . 댓글을 남길 때는 구체적으로 작성하여 상대방이 정확히 무엇을 언급하는지 알 수 있도록 하세요 . 코드를 제대로 이해했는지 확인하기 위해 명령줄이나 코드 공간에서 코드를 열어 실행해 보는 것도 도움이 될 수 있습니다 .
