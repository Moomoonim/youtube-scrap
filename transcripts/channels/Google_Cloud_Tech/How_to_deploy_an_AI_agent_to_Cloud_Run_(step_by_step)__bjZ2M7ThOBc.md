## 스크립트

[음악] AI 에이전트가 노트북에서만 실행되는 경우 , 이 비디오는 클라우드 런에 배포하는 방법을 정확하게 보여줍니다. 다음 몇 분 안에 Cloud Run, 에이전트 개발 키트 및 MCP 서버와의 보안 연결을 사용하여 로컬 AI 에이전트를 깔끔하고 안정적인 방식으로 인터넷에 배포하겠습니다. 이 비디오를 끝까지 시청하시면 클라우드 런에서 실행되는 동물원 투어 가이드 에이전트와 클릭해서 사용해 볼 수 있는 공개 URL을 갖게 될 것입니다 . 또한 프로젝트 설정, 필요한 코드, 모든 것을 작동시키는 데 필요한 서비스 계정 및 권한, 컨테이너를 빌드하고 배포하는 단일 명령, 그리고 브라우저에서의 간단한 테스트까지 단계별로 살펴보겠습니다. 코드를 다루기 전에 Cloud Run이 어떻게 작동하는지 간단히 살펴보겠습니다. 팝업 스토어라고 생각하면 됩니다. 요청이 도착하면 앱이 실행되는 컨테이너가 열립니다. 트래픽이 급증하면 Cloud Run은 더 많은 복사본을 생성합니다. 상담원에게 아무도 방문하지 않으면 사용량이 0으로 줄어들고 유휴 시간에 대한 비용은 전혀 발생하지 않습니다. 특정 포트에서 수신 대기하는 컨테이너를 가져오면 Cloud Run이 확장, 로드 밸런싱, TLS, 로그 및 ID 관리 등 나머지 모든 것을 처리합니다. 그래서 AI 에이전트를 배포하는 데 매우 적합합니다. 에이전트는 누군가가 실제로 사용하기 전까지 조용히 대기할 수 있으며 , 사용 요청이 들어오면 깨어나서 요청을 처리하고 다시 대기 상태로 돌아갑니다. 계획은 다음과 같습니다. ADK를 사용하여 에이전트용 간단한 Python 프로젝트를 만들고 도구를 빌드하겠습니다. 저희가 MCP 서버를 가리키도록 설정해 드리겠습니다. 먼저 서비스 계정에 적절한 권한을 설정한 다음, 단일 배포 명령을 실행하여 배포하겠습니다. 완료되면 에이전트가 포함된 URL을 얻게 됩니다 . 이 프로젝트를 따라가려면 두 가지가 필요합니다. Google Cloud 계정과 프로젝트 ID가 필요합니다. Google Cloud 계정을 만들려면 cloud.google.com으로 이동하세요. 아래 설명란에도 링크가 있으니, 그 링크를 클릭하시면 간편하게 무료 계정을 만드실 수 있습니다. 무료 계정을 만드신 후에는 프로젝트 ID를 생성하셔야 합니다. 프로젝트 ID를 생성하는 방법은 이 실습의 3단계에서 확인할 수 있습니다 . 설정하는 방법은 실제로 매우 간단하며 , 이 코드랩에서 설정 방법을 자세히 안내해 드리겠습니다. 이 코드랩은 Cloud Run에서 MCP 서버를 사용하는 ADK 에이전트를 빌드하고 배포하는 프로젝트이며 , 이 비디오에서 설명하는 모든 단계는 이 코드랩의 모든 단계에 포함되어 있습니다. 이 자료는 여러분에게 아주 좋은 참고 자료가 될 것이며, 해당 코드랩 링크는 아래 설명란에서도 찾을 수 있습니다. 자, Google Cloud 계정을 만들고 프로젝트 ID를 설정했다면 Google Cloud 콘솔의 홈 페이지는 다음과 같이 표시될 것입니다. 이제 Cloud Shell을 열기만 하면 됩니다. 그러려면 오른쪽 상단 모서리에 있는 이 아이콘을 클릭하거나, 맨 위 검색창에 Cloud Shell을 입력하여 열 수 있습니다. 먼저 올바른 Google Cloud 프로젝트에 접속했는지 확인해 보겠습니다. Google Cloud 콘솔에서 Cloud Shell을 열고 다음 한 줄을 실행하여 프로젝트를 선택하세요. 다음으로, 에이전트 배포에 필요한 서비스를 활성화합니다 . 따라서 Cloud Run은 Cloud Build 및 Artifact Registry를 사용하여 컨테이너를 빌드하는 데 도움이 되는 Run API가 필요합니다 . 에이전트가 Vertex AI 모델을 호출하고 내부적으로 컴퓨팅 API를 사용해야 합니다. 이 단계는 프로젝트당 한 번만 수행하면 됩니다. 완료되면 " 성공적으로 완료되었습니다"라는 긴 작업 문자열이 표시됩니다. 다음으로 에이전트용 작업 공간을 생성합니다. 클라우드 셸에서 새 폴더를 만들고 해당 폴더로 이동합니다. 다음으로 requirements.txt 파일을 생성하세요. 컨테이너 크기를 작게 유지하고 콜드 스타트 ​​시간을 적절하게 하려면 컨테이너 내의 종속성을 최대한 단순하게 유지하세요. 이번 실습에서는 간단한 예제를 위해 ADK, 간단한 위키피디아 도우미, 그리고 LangChain 어댑터를 사용하겠습니다 . 필요한 모든 라이브러리를 붙여넣고 , 라이브러리 업데이트 시 예기치 않은 호환성 문제가 발생하지 않도록 버전을 고정해 두세요. 코드랩을 따라하고 있다면 다음 명령어를 정확히 사용하세요 . 이제 스크립트와 코드가 프로젝트와 ID를 찾을 수 있도록 몇 가지 환경 변수를 설정하세요. 또한 Cloud Run 서비스가 런타임에 사용할 전용 서비스 계정을 생성하게 됩니다. 해당 ID는 Vertex AI를 호출할 수 있는 권한이 필요하며 , MCP 서버가 보호되어 있는 경우 해당 서버를 호출할 수 있는 권한도 필요합니다 . 다음으로, 간단한 .env 파일을 만들어 보겠습니다. 여기에서 모델을 선택하고 MCP 서버 URL을 저장합니다. .env 파일을 사용하면 코드 에서 비밀 정보와 설정 파일을 분리할 수 있습니다 . Cloud Run에서는 민감한 항목을 구성할 때 환경 변수 또는 시크릿 관리자를 계속 사용해야 합니다. 하지만 이번 실습에서는 .env 파일을 사용하여 최대한 간단하게 진행하겠습니다. 이제 모델에 대한 한 줄을 추가하세요. 다음으로, 새 서비스 계정에 MCP 서버를 호출할 수 있는 권한을 부여하십시오. 많은 사람들이 이 부분을 잊어버리고, 이로 인해 403 오류가 많이 발생합니다. 따라서 에이전트가 MCP 서버를 안전하게 호출할 수 있도록 프로젝트에 run.invoker 역할을 바인딩해야 합니다. 이제 MCP 서버 URL을 .env 파일에 입력하세요. 설정이 완료되면 에이전트용 소형 패키지를 생성합니다. __init__.py 파일은 파이썬에게 이 디렉터리를 가져올 수 있음을 알려주고, ADK에게 에이전트 디렉터리의 위치를 ​​알려줍니다. 자, 다음으로 패키지가 에이전트 모듈을 노출하도록 import 문을 하나 추가해 보겠습니다. 작업이 완료되면 새 agent.py 파일을 엽니다 . 이 파일에서 에이전트를 작은 단위로 나누어 구축해 나가겠습니다. 첫 번째 단계로 모든 가져오기 및 로깅 세부 정보를 추가했습니다. 다음으로 환경 변수를 불러오고, 모델 이름을 선언하고, 도구를 설정하겠습니다. 자, 우선 이 코드를 복사해서 agent.py 파일에 붙여넣으세요. 이 에이전트는 세 가지 기능을 사용합니다. 첫째, 사용자의 첫 번째 프롬프트를 공유 상태에 저장하여 당사의 동물원 투어 가이드 에이전트 프레임워크의 일부인 다른 에이전트에 전달할 수 있도록 하는 간단한 함수입니다. 둘째, 안전한 Zoo MCP 서버에 연결되는 MCP 도구 세트입니다 . 마지막으로 일반적인 사실을 찾아볼 수 있는 위키피디아 검색 도구도 있습니다. 그럼, 먼저 가져 오기와 환경 설정부터 시작하겠습니다. 여기서 클라우드 로깅은 단순히 깔끔함을 위한 것이 아니라 , 에이전트의 출력 내용과 경고가 Google Cloud Console의 로그 탐색기에 표시되도록 하기 위한 것입니다 . 이는 테스트 중에 정확히 필요한 기능입니다. 다음으로, 방문자의 첫 번째 입력 내용을 저장할 수 있는 간단한 도우미 도구를 추가하세요 . 이것은 안내자와 연구자 사이를 이어주는 접착제 역할을 합니다. 이 기능은 공유 도구 컨텍스트의 상태 사전(context.state dictionary)에 기록되므로 체인의 다음 에이전트가 이를 읽을 수 있습니다. 그러니 최대한 간단하고 명확하게 작성하세요. 그다음에는 MCP 도구를 설정하겠습니다. MCP 서버 URL은 이전에 생성한 .env 파일에서 확인할 수 있습니다. 프로덕션 환경에서 MCP 서버에 인증이 필요한 경우(필수 사항임) , Cloud Run 사용자 그룹에 대한 서명된 ID 토큰을 가져와 각 호출 시 베어러 토큰으로 전달해야 합니다. ADK의 스트리밍 가능한 HTTP 연결 매개변수를 사용하면 이 작업이 간단합니다. 마지막으로, LangChain 어댑터를 활용하고 위키피디아 쿼리를 래핑하는 일반 지식 도구를 추가하여 이 파일이 검색 클라이언트 코드보다는 오케스트레이션에 집중할 수 있도록 합니다. 필요한 도구가 준비되었으니, 다음으로 두 명의 전문 에이전트를 정의하겠습니다. 첫 번째는 연구원 에이전트입니다. 이 프로그램은 사용자의 상태 메시지를 읽어 Zoo MCP 서버를 사용할지, 아니면 저희가 개발한 위키피디아 도구를 사용할지 선택합니다. 그리고 만약 두 가지 방법을 모두 사용해야 한다면, 상담원이 당신에게 묻는 질문에 답하기 위해 그렇게 할 것입니다. 지침은 스타일보다는 행동에 초점을 맞춰 간결하고 구체적으로 작성하겠습니다. 또한 다음 담당자가 결과를 가져올 수 있도록 출력 키를 제공할 예정입니다 . 우리가 만들 두 번째 에이전트는 발표자 에이전트입니다. 이 프로그램은 도구를 직접 언급하지는 않지만, 기본적인 조사 결과를 바탕으로 동물원에 특화된 내용으로 시작하여 흥미로운 일반적인 사실들을 덧붙여 친절한 답변을 작성해 줍니다. 다음으로, 순차 워크플로 에이전트를 사용하여 이 두 가지를 하나의 간단한 워크플로로 묶겠습니다. 이 프로그램은 리서처를 실행한 다음 포맷터를 실행하고, 각 단계를 거치면서 상태를 전달합니다. 나중에 루프 에이전트나 병렬 에이전트를 활용하고 싶다면 이를 더욱 복잡한 패턴으로 발전시킬 수 있습니다 . agent.py 파일의 마지막 부분에는 정말 중요한 내용인 루트 에이전트를 추가하겠습니다. 이것이 새로운 대화의 시작점이며 , ADK가 기대하는 바이기도 합니다. 방문객을 맞이하고, 사용자가 제공하는 간단한 보조 도구를 사용하여 방문객의 첫 메시지를 상태에 저장한 다음, 해당 컨텍스트를 우리가 만든 워크플로에 전달합니다 . 인사말을 별도로 분리해 두면 디버깅 시 로그에서 흐름을 쉽게 파악할 수 있습니다. 자, 여기까지가 agent.py 파일에 있는 코드의 전부입니다 . 이제 배포된 서비스가 실제로 모델을 호출할 수 있도록 권한을 설정하십시오. 먼저 빌드 중에 모델 값이 보이도록 .env 파일을 셸에 불러온 다음, 서비스 계정에 Vertex AI 사용자 역할을 부여하겠습니다 . 이 역할이 없으면 서비스는 정상적으로 시작되지만 런타임 시 권한 오류로 인해 모델 호출이 실패합니다. 이제 배포 준비가 완료되었습니다. ADK에는 많은 작업을 자동으로 수행해주는 배포 명령어가 포함되어 있습니다 . 이 기능은 소스 코드에서 컨테이너를 빌드하고, 아티팩트 레지스트리에 푸시하고, 적절한 기본 설정을 사용하여 Cloud Run 서비스를 생성합니다. 또한, CLI를 전역적으로 설치할 필요가 없도록 UVX를 활용할 예정입니다. 빠른 테스트를 위해 ADK 웹 UI를 실행할 수 있도록 프로젝트, 지역, 원하는 서비스 이름과 함께 UI라는 플래그를 전달하세요 . 또한, 실행할 서비스 계정을 지정하고 나중에 찾을 수 있도록 서비스에 레이블을 지정하십시오 . 몇 가지 안내 메시지가 표시될 겁니다. Cloud Run에서 cloud-run-source-deploy라는 이름의 저장소를 생성할지 묻습니다. 아직 저장소가 없으면 생성하지 마세요. 그러니, 예라고 하세요. 인증 되지 않은 접근을 허용할지 묻는 메시지가 나타날 수 있습니다. 이번 실습에서는 어디서든 링크를 클릭할 수 있도록 '예'를 선택해 주세요 . 실제 운영 환경에서는 서비스를 비공개로 유지하고 인증된 프록시 또는 백엔드 통합을 통해 서비스를 제공하는 것이 좋습니다. 명령 실행이 완료되면 다음과 같은 형식의 URL이 출력됩니다 . 저분이 바로 담당 상담원입니다. 클릭하고 ADK의 웹 UI를 사용해 보세요. 먼저 오른쪽 상단 모서리에서 토큰 스트리밍을 켜서 모델의 사고 과정을 실시간으로 관찰할 수 있습니다.
