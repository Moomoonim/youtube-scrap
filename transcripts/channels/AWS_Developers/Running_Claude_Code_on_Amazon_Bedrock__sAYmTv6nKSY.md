## 스크립트

전 세계 개발자들이 앤스로픽 의 클로드 코드(Claude Code)와 같은 AI 코딩 도구가 일상 업무를 어떻게 변화시켰는지 이야기하고 있으며, 그 결과 그 어느 때보다 많은 제품을 출시하고 있습니다. 하지만 일부 개발자는 데이터 상주, 청구 또는 규정 준수 문제로 인해 이러한 도구를 사용할 수 없습니다. 이제 Amazon Bedrock을 통해 AWS 내에 호스팅된 모델을 사용하여 Claude Code를 실행할 수 있습니다 . 즉, Claude Code를 사용하면서도 보안, 규정 준수 및 비용 관리 요구 사항을 충족할 수 있습니다. 주요 이점 중 하나는 Claude Code의 모든 기능을 활용할 수 있고 데이터가 AWS 인프라를 벗어나지 않는다는 것입니다. 이 영상에서는 아마존 베드락에서 클로드 코드(Claude Code)를 실행하는 방법에 대한 전체 과정을 안내해 드리겠습니다. Claude Code가 무엇인지, Bedrock에서 실행해야 하는 이유, 환경 설정, 지역 간 추론 및 프롬프트 캐싱에 대해 설명하겠습니다. 다음으로는 세부적인 비용 추적, 콘텐츠 필터링을 위한 Amazon Bedrock Guardrails , 그리고 새로운 Amazon Bedrock Mantle 엔드포인트를 포함하여 Bedrock에서 Claude Code를 실행하는 팀에 유용한 몇 가지 사항을 살펴보겠습니다. 자, 그럼 시작해 볼까요? 아직 Claude Code를 사용해 보지 않으셨다면, Claude Code는 코드베이스를 읽고, 파일을 생성 및 편집하고, 명령을 실행하고, MCP 및 명령줄을 통해 외부 도구와 통합하는 에이전트 기반 코딩 도구입니다 . 앱에 새로운 기능을 추가하거나 버그를 수정하는 것과 같은 기본적인 사용 사례가 있지만, 그 활용 범위는 훨씬 더 넓습니다. 팀에 특화된 반복 가능한 워크플로를 정의하기 위해 사용자 지정 스킬을 만들 수 있습니다. 예를 들어 애플리케이션 구축, 테스트 작성 또는 마이그레이션 처리 방법 등을 정의할 수 있습니다. 또한 하위 에이전트와 에이전트 팀을 생성하여 문제의 서로 다른 부분을 병렬로 처리할 수 있습니다. 프로젝트의 여러 부분을 조율하기 위해 여러 개의 Claude Code 인스턴스를 사용하는 것도 흔한 일입니다. 그렇다면 왜 아마존 베드락에서 실행해야 할까요? 몇 가지 설득력 있는 이유가 있습니다. 첫째, 그리고 이는 기업 팀에게 가장 중요한 점인데, 여러분의 데이터는 AWS를 절대 벗어나지 않습니다. AWS는 사용자의 데이터, 코드 또는 프롬프트를 기반으로 모델을 학습시키지 않으며, 해당 정보를 모델 제공업체와 공유하지도 않습니다. Bedrock 버전을 실행할 경우, 일반적으로 Anthropic Games Store로 전송되는 모든 원격 측정 데이터( 피드백 데이터 포함)는 기본적으로 비활성화됩니다. 데이터 상주 요건이 엄격한 경우라면 바로 이것이 필요합니다. 둘째, Bedrock에 포함된 익숙한 인프라 와 보안 및 규정 준수 도구를 활용할 수 있습니다 . 로깅에는 IAM 자격 증명, Amazon CloudWatch, AWS CloudTrail을 사용할 수 있으며, 콘텐츠 필터링 또는 개인 식별 정보(PII) 삭제에는 Bedrock Guardrails를 사용할 수 있습니다. 이미 AWS를 사용하고 있는 조직이라면, 이 기능은 기존 시스템에 바로 통합됩니다 . 셋째, 사용량은 기존 AWS 청구 계정에 청구됩니다. 이 기능을 사용하기 위해 별도의 공급업체 계약이 필요하지 않습니다. 이 영상 후반부에서는 팀별 또는 사용자별 비용 분석 내역에 액세스하는 방법과 비용 할당 태그를 사용하여 지출에 대한 세부적인 정보를 얻는 방법을 다룹니다. 좋습니다, 그럼 이게 실제로 어떻게 작동하는지 보여드리겠습니다. 클로드 코드와 베드락이 대화하게 하려면 먼저 몇 가지 준비가 필요합니다. Claude Code와 AWS 명령줄 인터페이스가 설치되어 있어야 하고, AWS 계정과 Bedrock 모델을 호출할 수 있는 IAM 권한이 필요합니다 . Claude Code를 시작하기 전에 Anthropic의 API 대신 Bedrock을 통해 라우팅하도록 Claude Code에 지시하는 설정을 구성해야 합니다 . Claude를 실행한 다음 /config를 입력하여 설정 파일을 열거나, .claude/settings.json 경로에 있는 설정 파일을 직접 편집하여 이 작업을 수행할 수 있습니다. 저희는 이 파일을 직접 편집할 예정입니다. 설정에 다음 값을 추가하세요. 첫 번째 변수인 Claude code use bedrock equals one은 Claude code에게 기반암을 가리키도록 지시하는 스위치입니다. 그리고 AWS 리전은 어느 리전으로 요청을 보낼지 알려줍니다. 이 두 가지는 베드락 에디션을 실행하는 데 필요한 최소한의 설정입니다. 또한 특정 버전을 고정할 수 있도록 몇 가지 모델 변수를 설정해 두었으므로 사용 중인 모델을 정확하게 제어할 수 있습니다. AWS Bedrock에서 유효한 모델 ID를 찾는 방법을 알고 싶다면 ` AWS Bedrock list foundation models by provider Anthropic` 명령어를 실행하면 됩니다. 이 함수는 설정에서 참조할 수 있는 기본 모델 목록을 반환합니다. 이 정보는 settings.json 파일에 저장되므로, 사용자가 선택한 설정은 터미널 세션 간에 자동으로 유지됩니다. 터미널을 열고 `Claude`를 입력할 때마다 Bedrock에 연결되며, 이미 구성된 로컬 AWS 자격 증명을 사용하여 Bedrock에 대한 보안 연결을 설정합니다. 이는 표준 AWS SDK 자격 증명 체인을 따릅니다. 이미 AWS 명령줄에 사용할 자격 증명이 있는 경우 Claude 코드는 해당 자격 증명을 자동으로 인식합니다 . IAM Identity Center를 사용 중인 경우 먼저 프로필로 로그인하기만 하면 됩니다. 요즘 AWS에서 인증하는 가장 일반적인 방법 중 하나는 IAM Identity Center를 사용한 싱글 사인온(SSO)이며, 이 방식은 임시 자격 증명을 사용합니다. 즉, 크레딧은 언젠가는 만료되는데, 보통 한 시간 정도 후입니다. 하지만 코딩 세션 도중에 토큰이 만료되더라도 바로 로그아웃되는 것은 아닙니다. Claude 코드는 새 자격 증명을 자동으로 가져오는 기능을 지원합니다. 이를 설정하려면 다음 JSON 코드 조각을 설정에 추가해야 합니다 . 이 설정은 AWS 인증 갱신 설정을 정의하여 Claude 코드가 자격 증명이 만료될 때 새 자격 증명을 가져올 수 있도록 합니다. 백그라운드에서 새로 고침이 진행되어 모든 것이 계속 작동하도록 유지합니다. 사용자가 정의한 설정은 구성 파일이 있는 위치에 따라 서로 다른 범위에 적용될 수 있습니다 . 대부분의 개발자에게 중요한 세 가지 요소가 있습니다 . 사용자 범위는 홈 디렉터리의 .cloud/settings.json 경로에 있는 개인 설정 파일입니다. 여기에 입력하는 내용은 컴퓨터의 모든 프로젝트에 적용됩니다. 기본적으로 기본 환경 변수와 모델 설정이 저장되는 곳입니다. 프로젝트 범위는 해당 위치의 저장소에 있습니다. 이 내용은 프로젝트 에 반영되므로, 여기에 입력하는 모든 내용은 프로젝트의 모든 협력자에게 적용됩니다 . 로컬 범위는 동일한 저장소의 .cloud/settings.local.json에 있지만 자동으로 무시됩니다. 그러니까, 그건 해당 프로젝트에서만 당신에게 적용되는 겁니다 . 개인 설정 변경에 사용하시면 됩니다. 여러 범위에 동일한 설정이 있는 경우 상위 범위의 설정이 우선합니다. 따라서 팀 전체가 클라우드 코드를 사용하는 경우 프로젝트 범위를 사용하여 협업자 간에 모델 버전과 지역을 표준화하고 해당 파일을 커밋하여 팀 구성원 모두가 동일한 설정을 공유하도록 할 수 있습니다. 다양한 설정을 원하는 개발자는 로컬 스코프를 사용합니다. 이제 `cloud` 명령어를 실행하여 클라우드 코드를 시작해 보겠습니다 . Amazon Bedrock이 공급자로 표시되는 환영 화면이 나타날 것입니다 . 자, 본격적인 빌드를 시작하기 전에 AWS용 에이전트 툴킷 플러그인을 추가해 보겠습니다. 이를 통해 Cloud Code는 도구와 모범 사례에 접근하여 AWS에서 애플리케이션을 더욱 효율적으로 구축할 수 있습니다 . 설치하려면 `/plugin marketplace add AWS Agent Toolkit` 명령어를 실행하고 Enter 키를 누르면 됩니다. 그다음에는 `/reload plugins` 명령어를 실행하고 엔터 키를 누르면 플러그인을 사용할 준비가 완료됩니다. 이제 우리는 클라우드 에게 서버리스 URL 단축기를 생성하고 CDK와 Python을 사용하여 AWS에 배포하는 것과 같은 작업을 요청할 수 있습니다 . 그런 다음 엔터 키를 누르면 실행됩니다. 그러면 에이전트 루프가 실행되어 이를 구축하고 필요에 따라 권한을 요청하며 , 모든 추론 작업은 백그라운드에서 Bedrock을 대상으로 실행됩니다. 클라우드 코드를 많이 사용하는 사용자에게 흔히 발생하는 문제 중 하나는 스로틀링입니다. 이는 API 용량 제한이나 사용량 제한에 도달하여 오류가 발생하는 현상입니다. Bedrock에서 제공하는 지역 간 추론 기능은 이러한 문제를 완화하는 데 도움이 될 수 있습니다. 지역 간 추론을 사용할 때 Bedrock은 사용 가능한 용량이 있는 지역으로 요청을 자동으로 라우팅합니다. 이를 위해 클라이언트 측 로드 밸런싱을 구축할 필요가 없으며, 가격도 동일하게 유지됩니다. 처리량이 훨씬 더 많아질 뿐입니다. 설정 파일에 입력한 모델 ID를 통해 설정됩니다 . 교차 지역 추론 프로파일에 대한 ID를 제공하면 Bedrock이 나머지를 처리합니다. 접두사 없이 단일 지역을 가리키는 모델 ID를 제공하면 해당 지역으로 고정되어 해당 지역의 처리량으로 제한됩니다. 지역 내 모델 ID는 지연 시간에 민감한 워크로드에 대해 가장 가까운 지역으로 라우팅해야 하거나 확정적인 라우팅이 필요한 경우에 가장 적합합니다. 지역 간 추론 프로필을 사용하려면 지리적 지역 간 라우팅을 위한 모델 ID를 제공해야 합니다. 지리적 지역 간 추론 프로필에는 US, EU 또는 APAC와 같은 접두사가 붙습니다. 귀하의 요청은 해당 지역 내 어디든 전송될 수 있습니다 . 따라서 미국 접두사를 사용하면 트래픽이 미국 지역 내에 머무르지만, 한 지역이 혼잡해지면 다른 지역으로 이동할 수 있습니다 . 대부분의 팀에게 제가 추천하는 기본 설정입니다. 다음으로는 전역 접두사를 사용하는 전역 교차 지역 추론 프로필이 있습니다 . 이를 통해 귀하의 요청은 해당 모델에서 지원되는 전 세계 모든 지역으로 라우팅되어 최대 처리량을 얻을 수 있습니다. 방금 보여드린 설정 방법을 따라하셨다면 , 모델 ID가 미국 접두사로 시작하므로 이미 지리적 교차 지역 추론을 사용하고 계신 것입니다. 지금까지 보여드린 모든 내용은 표준 Bedrock 런타임 엔드포인트를 사용합니다. 그 방법도 작동은 하지만, AWS에서 향후 사용을 권장하는 Amazon Bedrock Mantle이라는 새로운 엔드포인트가 있으며 , 그 이유는 몇 가지가 있습니다. Mantle은 Anthropic 메시지 API와 같은 네이티브 API 형태를 통해 모델에 서비스를 제공하며, 향상된 처리량 처리, 별도의 입력 및 출력 토큰 할당량, 성능 계층을 지원합니다. 모두 동일한 Bedrock 인프라 및 엔터프라이즈 제어 기능을 기반으로 합니다. 그래서 방금 설명드린 모든 혜택과 기능을 그대로 누리실 수 있습니다. Mantle은 또한 운영자 접근이 전혀 불가능한 구조로 설계되었습니다. 즉, Bedrock을 실행하는 운영자인 AWS조차도 추론 데이터에 접근할 수 없다는 뜻입니다 . 이 시스템은 백도어, 관리자 권한 재정의 또는 지원 엔지니어가 사용자의 프롬프트를 불러올 수 없도록 설계되었습니다. 기술적인 해결책이 아예 존재하지 않습니다. 따라서 Claude를 통해 Bedrock으로 코드를 전송할 때 해당 대화는 전적으로 AWS 계정 내에서 이루어집니다 . Mantle은 계정별 엄격한 제한 대신 고급 스케줄링 및 작업 대기열 아키텍처를 사용합니다 . 이렇게 하면 용량이 부족할 때 요청이 거부되는 대신 잠시 대기열에 쌓이게 됩니다 .
