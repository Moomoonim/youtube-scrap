## 스크립트

안녕하세요. 카카오 SA 개발 파트에서 키어드 광고 DSP 서버 개발을 담당하고 있는 제노스라고 합니다. 6억간의 광고 데이터 마이그레이션이라는 주제로 발표를 시작하겠습니다. 키워드 광고 도메인에 대한 개념이 생소하신 분들을 위해서 먼저 해당 도메인에 대한 설명을 간략히 드리겠습니다. 키워드 광고는 광고 대상 키워드 소재로 구성됩니다. 옆에 보시는 그림처럼 포털에서 특정 키워드를 검색했을 때 나오는 광고가 키워드 광고에 해당합니다. 여기서 키워드는 말 그대로 광고를 나오게 할 텍스트를 의미합니다. 비즈 채널은 특정 키워드에 대해서 광고하려는 대상을 나타냅니다. 광고에 포함된 이미지, 타이틀, 설명 등을 소재라고 합니다. 광고주가 광고를 구매할 수 있는 플랫폼을 디맨드 사이드 플랫폼 줄여서 보통 DSP라고 합니다. 광고들이 DSP를 통해 구매한 광고는 특정 지면에 노출될 기회를 얻게 됩니다. 키워드 광고의 경우 하나의 키워드에 대해 여러 개의 광고가 노출될 수 있습니다. 그래서 같은 키워드로 광고를 구매한 광고주들끼리 입체가 변경이나 소재 품질을 높이는 방식으로 경쟁하게 됩니다. 또한 광고주들은 광고가 노출될 확률을 높이기 위해 다양한 키워드들을 구매합니다. 연관된 키워드에 동의어 상품일 경우에는 상품 코드나 종류 등 수많은 키워드들을 사용하게 되면서 DSP에서 관리해야 하는 키워드수가 급증하게 됩니다. 많은 키워드들을 좀 더 편하게 관리할 수 있도록 DSP에서는 광고 그룹과 캠페인 개념을 도입했습니다. 광고 그룹은 여러 개의 키워드와 소재를 포함하고 광고 노출 전략, 입찰 전략 등을 설정할 수 있습니다. 캠페인은 여러 개의 광고 그룹을 포함하고 광고 집행에 필요한 일산 등을 설정할 수 있습니다. 다음으로 마이그레이션 프로젝트의 배경에 대해 설명드리겠습니다. 기존 키워드 광고 DSP는 앞서 말씀드린 기능들과 함께 광고, 심사, 보고서 관련 기능들이 하나의 큰 모널리식ic 아키텍처로 구현되어 모든 데이터를 하나의 DB로 관리했습니다. 애드 서버를 제외하고는 많은 기능들이 한 군데에 섞여 있어서 개별적으로 새로운 기능을 추가하기 어려웠고 유지보수도 용의하지 않았습니다. 여기서 에드 서버는 DSP의 데이터를 다른 장소로 동기화하여 특정 키워드에 대해서 노출될 수 있는 광고 리스트를 응답해 주는 역할을 합니다. 에드 서버까지 포함해서 DSP라고 할 수 있는데요. 이번 발표에서는 광고주와 직접적인 상호 작용을 하는 부분만을 DSP라고 보겠습니다. 다시 본론으로 돌아가면 유지보수와 새로운 기능 개발의 단점을 해소하고자 신규 DSP는 기능들을 분리하여 마이크로비스 아키텍처로 구현되었습니다. 광고주, 마케터 등과 관련된 사항들은 광고 계정 플랫폼을 통하고 광고주가 구매한 키워드 광고에 적합한지 심사하는 기능도 광고 심사 플랫폼을 통해 제공하게 되었습니다. 보고서와 관련된 데이터도 다른 플랫폼으로 분리되어 DSP에서 필요한 정보를 AP 형태로 요청하도록 바뀌었습니다. 신규 DSP의 루필과 접속 방식도 같이 변경되면서 기존 광고주들의 이탈이 우려되었습니다. 또한 기존 DSP와 신규 DSP의 광고를 함께 노출시키는 것은 불가능하여 개편 후에 신규 DSP에서 구매된 광고만 노출이 가능했습니다. 신규 DSP에서 광고주들이 빠른 CL의 광고들을 다시 구매하지 않으면 비즈니스 측면에서 큰 손실이 예상되었습니다. 하지만 광고주들 입장에서는 기존 DSP에서 구매했던 다수의 광고들을 신규 DS에서 재구매하는 작업이 큰 부담이 됩니다. 그래서 매출 손실과 건구주들의 이탈을 최소화하기 위해 마이그레이션 기능을 제공하기로 했습니다. 그럼 마이그레이션 규모는 어느 정도였을까요? DB 로우스 기준으로 6억권이고 대부분 키워드 데이터입니다. 합쳐져 있었던 기능들이 분리되었기 때문에 각 플랫폼에도 필요한 형태로 데이터를 전달해야 합니다. 마이그레이션을 진행할 수 있는 대략적인 일정은 어땠을까요? 계획되었던 마일스토는 이렇습니다. 한 달간 마이그레이션 신청을 받고 신청 기간이 종료되면 종료 시점의 데이터를 기존 서비스 중단 없이 주말 동안 마이그레이션 해야 합니다. 주말이 끝나면 마이그레이션된 데이터를 확인할 수 있도록 신규 DSP를 오픈합니다. 데이터 확인 기간 종료 후 광고가 바로 노출될 수 있도록 다른 플랫폼으로의 데이터 전달도 완료되어야 합니다. 정리하면 6억권의 데이터를 기존 서비스 중단 없이 2일 안에 마이그레이션 해야 합니다. 와이그레이션이 지연될수록 광고 노출이 늦어지기 때문에 여유 있게 진행할 수 있는 대책이 필요했습니다. 그 당시에 당장 떠오르는 방법은 없어서 마이그레이션을 위한 프로토타이핑을 해보며 나은 방법들을 계속 찾아보기로 했습니다. 주어진 신규 DSP DB는 최대 5,000 TPS로 처리가 가능하여 6억권 처리에 33시간 정도 소요된다고 예상할 수 있었습니다. 중간에 장애가 발생할 경우 더 걸릴 수 있었지만 우선은 신규 DSP DV로 데이터를 마이그레이션 하는 기능부터 프로토타이핑하기 시작했습니다. 가장 개수가 적은 비즈 채널부터 캠페인, 광고 그룹, 소재, 키워드 순서대로 진행하는 시나리오를 선택했습니다. 여기서 두 DSP의 도메인 구조 차이로 인해 크게 세 가지의 이슈가 발생했습니다. 첫 번째로 기존 DSP의 캠페인과 비지 채널은 직접적인 연관 관계가 없어서 비지 채널 마이그레이션이 진행된 후에 캠페인만 마이그레이션이 불가능했습니다. 그래서 캠페인 광고 그룹을 합쳐서 마이그레이션 하기로 하고 기존 DSP에서 광고 그룹을 먼저 조회해서 연결된 캠페인을 역으로 같이 조회했습니다. 두 번째로 신규 DSP에서는 소재 라이브러리라는 개념을 도입하여 광고 그룹과의 직접적인 연관 관계가 사라졌습니다. 이로 인해 비지 채널별로 소재의 중복을 제거하는 과정이 추가되었습니다. 세 번째로 아이디 생성 이슈가 있었습니다. 마이그레이션을 통해 생긴 데이터와 신규로 생성된 데이터의 아이디가 겹칠 가능성이 있었습니다. 다행히 기존에는 오토인크리먼트 방식을 사용했었고 신규 DSP에서는 타임스탬프 기반으로 아이디를 생성화했습니다. 이때 기존 DSP의 도메인별 아이디의 최댓값의 자릿수보다 신규 DSP에서 훨씬 더 큰 자릿수로 아이디가 생성되었기 때문에 기존 도메인의 아이디를 그대로 신규 플랫폼의 데이터에 사용할 수 있었습니다. 하지만 여기서 캠페인은 그대로 사용할 수 없었습니다. 신규 DSP의 캠페인은 하나의 비지 채널만 가져야 합니다. 이로 인해 캠페인과 비지 채널의 조합으로 캠페인을 새로 만드는 로직이 추가되었습니다. 그래서 슬라이드에 나온 예시에서는 두 개였던 캠페인이 세 개로 늘어나고 신규 DSP의 캠페인 명과 아이디 모드 기존 DSP의 비슷 채널과 캠페인 조합을 통해 만들어졌습니다. 이렇게 기존 DSP의 비지 채널과 캠페인 조합을 통해 신규 캠페인의 아이디 생성 이슈도 해결되었습니다. 다른 도메인 구조로 인한 이슈를 해결하고 API를 통해서 마이그레이션을 시작하려고 보니 한 번에 트랜잭션으로 모든 작업을 처리하기에는 무리였습니다. 그래서 트랜잭션을 여러 단계로 분리했습니다. 먼저 도메인별로 트랜잭션을 분리하고 도메인 내에서도 페이징을 통해 특정 개수만큼 순차적으로 마이그레이션 되도록 트랜잭션을 분리했습니다. 소재의 경우에는 중복 제거 처리 로직으로 인해서 베이징에 불과하여 제외시켰습니다. 그리고 마이그레이션 신청을 받은 광고 계정 기준으로 진행하면 되기 때문에 광고 계정별로 트랜잭션을 시작했습니다.이를 이를 통해 여러 개의 광고 계정에 대해서 동시에 처리가 가능해졌고 각 계정별로 마이그레이션의 진행도 확인하기 용이해졌습니다. 분리된 트랜잭션들이 순차적인 이벤트를 통해 처리될 수 있도록 마이그레의 전용 카프카를 연동하게 되었습니다. API를 통한 마이그레이션 요청도 광고정 단위로 요청할 수 있도록 수정했습니다. 카프카를 통해 이벤트를 처리하는 과정에서도 이슈가 발생했습니다. 하나의 인스턴스에서 여러 계정의 마이그레이션이 동시에 진행되는 모습을 원했지만 실제로는 하나의 광고 계정 이벤트만 처리되고 있었습니다. 그리고 각각의 이벤트를 처리하는데 너무 오래 걸리면 다음 이벤트 소비가 늦어집니다. 이로 인해 이벤트 처리 시간에 타임아웃이 기본적으로 설정되어 있었기 때문에 트랜잭션을 분리했음에도 불구하고 타임아웃을 초과하여 예외도 종종 발생했습니다. 타임아웃과 병렬 처리 이슈를 해결하기 위해 별도의 쓰레드풀에서 이벤트를 받아서 처리하도록 수정했습니다. 다른 쓰레드가 이벤트를 처리했기 때문에 카프카 입장에서는 처리도 빨라진 것처럼 보여서 타임아웃을 늘릴 필요가 없었습니다. 이때 쓰레드들이 각각 트랜잭션을 수행해 DB 커넥션 풀의 최대 크기를 늘려주고 더 많은 DB 리소스를 사용하게 되었습니다. 300개의 광고 계정에 대해서 동시에 마이그레이션을 진행했을 때 DB 리소스의 약 90% 정도를 사용하는 것이 확인되어이 이상으로 리소스를 쓰지 않도록 조절했습니다. 이벤트를 처리하는 도중에 에러를 어떻게 처리할지도 정리가 필요했습니다. 페이징 단위로 마이그레이션을 진행할 때 트랜잭션 전체를 실패 처리하기보다 한 페이지를 구성하는 엔티 하나하나씩 트랜잭션을 생성해서 마이그레이션을 수행했습니다. 실패한 경우는 재처리가 가능하도록 로그만 남기고 페이징 단위에 마이그레이션은 완료 처리한 후 다음 이벤트를 발행하여 처리를 계속 진행했습니다. 로깅한 에러는 사내에서 제공되는 AEM인 매트릭스를 통해 확인이 가능하게 세팅했습니다. 기존 DSP 로직과 DB 데이터의 차이가 에러의 주의 원인이었습니다. 기존 DSP 로직에서는 널이 불가하지만 기존 DB상에는 널로 들어가 있는 경우를 예로 들 수 있습니다. 이렇게에 여러 프로토타입의 로직을 수정하여 특정 광고 계정에 대해 처음부터 마이그레이션을 다시 진행하거나 특정 엔티만 재처리할 수 있도록 API를 추가하였습니다. 예외 상황에 대한 정책이 정리된 후 키워드 마이그레이션을 진행할 때 성능 이슈가 새로 확인되었습니다. 예상하긴 했지만 데이터가 너무 많아서 조회 속도가 느렸습니다. 우선 커리를 튜닝하고 JPA 네이티브 커리를 사용하여 조회하도록 수정했습니다. 네이티브 커리스음에도 불구하고 JPA에서 ORM을 위한 추가적인
