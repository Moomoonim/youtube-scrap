## 스크립트

코딩 도우미, 고객 지원 담당자 또는 연구 도우미와 같은 단일 작업 도구 이상의 것을 구축하려면 에이전트 실행 또는 세션 간에 대화 기록이 유지되어야 합니다. Strands에서는 세션 관리자가 이를 처리합니다. 내장된 파일 세션 관리자를 사용하여 세션 정보를 로컬 파일에 저장하는 것부터 시작할 수 있습니다. 지금까지 사용해 온 고객 서비스 예시를 통해 코드를 살펴보겠습니다. 파일 세션 관리자를 가져옵니다. 그다음 세션 관리자의 새 인스턴스를 생성하고 세션 ID와 스토리지 디렉터리를 전달합니다. 이 세션 ID는 특정 대화를 식별하는 데 사용되며, 저장 디렉터리는 파일이 저장될 위치입니다 . 실제 응용 프로그램에서는 이러한 값을 하드코딩하지 않습니다. 이러한 기능은 특정 사용자 세션에 맞춰 범위가 지정되어야 합니다. 그래야 두 사용자가 동일한 상담원을 사용하는 경우에도 대화 기록에 대해 서로 다른 독립적인 세션을 갖게 됩니다 . 하지만 지금은 배우는 단계이니 이 정도면 괜찮아요. 최종적으로는 실제 운영에 바로 적용 가능한 솔루션을 개발할 것입니다. 세션 관리자를 사용하려면 에이전트를 생성할 때 세션 관리자를 전달하기만 하면 됩니다 . 그게 전부입니다. 이제 모든 메시지와 상태 변화가 지정된 디렉터리에 자동으로 디스크에 저장됩니다. 한번 시도해 보겠습니다 . 그럼 이제 대화를 시작해 볼까요? "주문 반품 좀 도와주세요."라고 말하세요. 보시다시피 세션 디렉토리가 생성되었습니다. 그리고 이 화면을 열어보면 상담원 메시지가 들어오는 것을 확인할 수 있습니다. 그래서 메시지 0번을 열어보면 "주문 반품을 도와주세요"라는 내용이 보입니다. 첫 번째 메시지는 "반품 처리를 도와 드릴 수 있어서 기쁩니다 ."라는 답변입니다. 여기에는 메타데이터와 기타 여러 정보도 포함되어 있습니다. 이제 C001 1001이라고 말할 수 있는데, 이것은 고객 ID입니다. 그리고 이제 측면에 더 많은 메시지가 추가되고 있습니다 . 그 외에도 세션 ID가 포함된 session.json 파일과 에이전트의 내부 상태 정보가 포함된 agent.json 파일과 같은 정보도 포함되어 있습니다. 채팅을 계속하면 여기에 있는 파일들이 계속 채워질 거예요 . 그러면 나갈 수 있습니다. 이제 동일한 파일 세션 관리자 구성으로 스크립트를 다시 시작하면 에이전트가 자동으로 전체 대화 기록과 상태를 복원합니다. 내가 " 무엇을 도와주시는 건가요?"라고 물었을 때, 시스템은 메시지를 찾아서 나에게 알려줄 수 있어야 합니다. 네, 주문 반품에 대해 문의하셨다고 나와 있으니, 해당 메모리를 복구하는 데 성공한 것을 알 수 있습니다. 지속성은 세 가지 지점에서 자동으로 발생합니다. 에이전트가 초기화될 때 기존 세션 데이터를 불러옵니다. 메시지가 추가되면 영구 메모리에 기록됩니다. 그리고 각 호출 후에는 에이전트 상태와 대화 관리자 상태를 동기화합니다. 저장이나 플러시를 호출할 필요가 없습니다. Strands가 자동으로 처리해 줍니다. 내부적으로 세션 관리자는 훅 제공자(hook provider)로 구현됩니다. 즉, 지속성은 실제로 에이전트 수명 주기에 후크를 통해 추가된 또 다른 하네스 동작일 뿐입니다. 그러니까 훅에 대해 배우는 모든 것은 지속성, 플러그인, 그리고 제어 기능을 모두 작동시키는 데 필요한 기본 요소입니다. 그러니까 저게 로컬 파일 세션 관리자입니다. 하지만 에이전트를 어딘가에 배포할 준비가 되면 이 정보를 로컬 디스크가 아닌 다른 곳에 저장해야 합니다. Strands의 장점은 전체 앱을 다시 작성하지 않고도 다른 세션 관리자를 정의하고 구성 요소를 교체할 수 있다는 점입니다. 예를 들어, 대화 기록을 AWS의 객체 스토리지 서비스인 Amazon S3에 저장할 수 있습니다. 이미 AWS 계정과 S3 버킷이 설정되어 있는 경우 S3 세션 관리자를 정의할 수 있습니다. 직접 가져오는 겁니다. 세션 ID, 버킷 이름, 선택적 접두사 및 AWS 리전을 전달하여 S3 세션 관리자 인스턴스를 생성합니다. 그런 다음, 그 내용을 담당 에이전트에게 전달하면 완료됩니다. 이 기능을 사용하면 이전에 보았던 것과 동일한 세션 구조가 저장되지만, 이제 로컬 파일 시스템 대신 S3에 저장됩니다. 나중에 아마존 베드락 에이전트 코어 메모리를 설정하고 사용하는 방법을 보여드리겠습니다. 이는 단기적인 데이터 저장 이상의 기능을 제공하기 때문에 특히 흥미로운 부분입니다 . 또한 장기 기억 패턴을 지원하여 대화 요약, 사용자 선호도 및 의미론적 사실과 같은 정보를 원시 대화 기록에서 자동으로 추출하여 필요할 때 에이전트가 검색할 수 있도록 합니다. 이 내용은 나중에 에이전트를 클라우드에 배포할 때 사용하겠습니다. 하지만 가장 좋은 점은 통합 패턴이 여전히 동일하다는 것입니다. 세션 관리자만 교체하면 됩니다. 여기서 또 하나 중요한 차이점이 있는데, 사람들이 많이 혼동하는 부분이라고 생각합니다. 세션 관리자는 대화 지속성을 관리하지만, 실제 운영 환경에서는 상담원의 컨텍스트 정보가 대화 기록 외에도 다양한 소스에서 제공되는 경우가 많습니다 . 데이터베이스의 고객 데이터, RAG 시스템의 제품 정보, 장기 기억의 사용자 선호도 또는 외부 시스템의 운영 상태 정보. 이는 별도의 컨텍스트 소스이며, 원시 대화 기록 저장소와는 별개로 에이전트에 연결하는 도구를 사용하여 런타임에 동적으로 데이터를 검색할 수 있습니다. 다음으로는 다중 에이전트 시스템에 대해 다루겠습니다. 먼저 가장 간단한 오케스트레이션 패턴 중 하나인, 에이전트를 다른 에이전트의 도구로 활용하는 패턴부터 살펴보겠습니다.
