## 스크립트

Multiv vector search delivers impressive retrieval quality. We've seen how it captures fine grain semantic relationships that single vector embeddings miss. But before you deploy multi vector search in production, you need to understand its limitations. This lesson is about being realistic. We are going to examine the challenges head on. The index incompatibility, the memory explosion, and the computational overhead. Understanding these trade-offs will help you make informed decisions about when and how to use multivactor search. The good news, module 3 covers optimization techniques that address many of these challenges. But first, let's understand what we are dealing with. Hnsw hierarchical navigable small word graph needs two things to work. Symmetric distances and query independent neighbors. The distance from A to B must equal the distance from B to A and document nearest neighbors must stay fixed regardless of the query. Maxim breaks both assumptions. The direction matters. Maxim Q and D doesn't equal maxim D and Q. And nearest neighbors are query dependent. They change based on which query you're evaluating. The result, we can't build a static proximity graph that makes HNSW fast. Every query requires comparing against every document. For a million documents, that's a million maximum calculations per query. HNSW works by building static proximity graphs that connect similar documents. When a query comes in, instead of comparing against every document, you navigate through this graph, following edges to increasingly similar documents. You start somewhere and follow edges toward similar documents. This is why single vector search can scale to billions of documents with subsecond latency. Logarithmic scaling. Without HNSW, we are back to linear scans. Every document, every query for a million documents, that's a million maximum calculations per query. And each maxim isn't just one operation. It's computing similarities between all query tokens and all document tokens. Then finding maxima and summing. Let's do the math. Single vector search at 384 dimensions. It's 384 * 4 bytes per float equals 1,536 bytes about 1 and a half kilobytes per document. Multi vector with Colbert I 228 dimensions but 500 tokens per document is around 250 kilobytes per documents that's 167 times more storage for the same number of document the lower dimensionality per token doesn't compensate for the sheer number of vectors 1 mill documents 1.5 GB for single vector versus 250 GB for multi vector the difference is massive Now we are looking at specialized hardware and creative memory management strategies. This diagram shows the storage comparison visually. The gap widens as documents get longer and collection grow. Multiv vector representations may use roughly 170 times more memory than single vector models. But that depends on the model dimensionality. That's why optimizations become essential. These challenges are solvable. Quantization may reduce each vector size by even up to 32 times. Instead of floats, we can store numbers as integers or even bits. Pooling can reduce the number of vectors per document. Instead of keeping every token, we combine similar ones. Multi-stage retrieval uses fast single vector methods for initial retrieval, then applies maxim for reranking the top candidates. Mu vera creates HNSW indexable approximations of multivactor representations enabling fast approximate search. We cover all these techniques in detail in module 3.
