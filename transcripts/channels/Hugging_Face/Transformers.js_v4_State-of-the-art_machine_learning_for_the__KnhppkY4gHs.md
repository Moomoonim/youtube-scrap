We just released Transformers.js version&nbsp; 4 and it is huge. I don't think there's a&nbsp;&nbsp; single line untouched in the core library. Our&nbsp; brand new C++ WebGPU backend brings unmatched&nbsp;&nbsp; performance to your web applications as well&nbsp; as serverside JavaScript. This new backend&nbsp;&nbsp; also enables you to run much larger&nbsp; models like GP-OSS 20B in JavaScript.&nbsp;&nbsp; Keeping up with the latest research and trends.&nbsp; We've added support for many new architectural&nbsp;&nbsp; components. We have exciting new features like the&nbsp; ModelRegistry and improved login controls. And the&nbsp;&nbsp; code base has been completely restructured. It's&nbsp; more modular, more maintainable with faster builds&nbsp;&nbsp; and smaller bundles thanks to our new build&nbsp; system. Hi, I'm Nico and together with Joshua,&nbsp;&nbsp; we develop Transformers.js. Let's dive&nbsp; in and see what's new in version 4. [music] The biggest change is the new WebGPU runtime&nbsp; completely rewritten in C++. We worked closely&nbsp;&nbsp; with the ONNX runtime team to test it across all&nbsp; supported models and the results are incredible.&nbsp;&nbsp; Our previous WebGPU backend was JavaScript only&nbsp; accessing the browser's WebGPU API directly.&nbsp;&nbsp; that worked but locked us into the browser.&nbsp; The new C++ backend changes everything.&nbsp;&nbsp; WebGPU operations can now run in any language,&nbsp; Python, C++, C#, JavaScript, and all platforms,&nbsp;&nbsp; which means the same Transformers.js&nbsp; code now runs across browsers, node,&nbsp;&nbsp; bun, and deno. WebGPU acceleration is&nbsp; no longer browser only. Besides that,&nbsp;&nbsp; it also provides better operator coverage,&nbsp; better performance, and better accuracy. We've proven that it is possible to run&nbsp; state-of-the-art AI models in the browser, and&nbsp;&nbsp; now we're making them fast, even on constrained&nbsp; devices. We rethought our entire export strategy.&nbsp;&nbsp; For large language models, we reimplemented&nbsp; architectures operation by operation and&nbsp;&nbsp; leveraged fused kernels to maximize performance.&nbsp; One great example is the LFM 2.5 thinking,&nbsp;&nbsp; a 1.2 billion parameter LLM from LiquidAI that can&nbsp; now run in the browser at impressive token speeds. Transformers.js version 4 now supports&nbsp; over 200 different architectures,&nbsp;&nbsp; many exclusive to version 4. There are almost&nbsp; 3,000 compatible models on the Hugging Face hub,&nbsp;&nbsp; and that number grows every day. Here are&nbsp; some V4 exclusive models you should know&nbsp;&nbsp; about. TranslateGemma enables multilingual&nbsp; translations across 55 different languages. LFM2VL brings real-time video captioning to your&nbsp; browser. Quen 3.5 is a multimodal model that&nbsp;&nbsp; performs at the same level as GPT-4o from a couple&nbsp; of years ago. Voxtral realtime delivers streaming&nbsp;&nbsp; automatic speech recognition "running 100% locally&nbsp; in my browser powered by Voxtral Realtime and&nbsp;&nbsp; Transformers.js". And Chatterbox Turbo enables&nbsp; voice cloning with just 5 seconds of reference&nbsp;&nbsp; audio. You can even use paralinguistic tags like&nbsp; [chuckle] or [gasp] to enhance the realism of your&nbsp;&nbsp; generations. "It was over long before you realized&nbsp; you were playing my game." Behind the scenes,&nbsp;&nbsp; getting these models working in ONNX was&nbsp; a challenge. Take mamba layers. We had to&nbsp;&nbsp; reimplement them operation by operation. You&nbsp; can see this in action with Granite 4.0-H,&nbsp;&nbsp; IBM's hybrid models that use interleaved mamba and&nbsp; attention layers or multi head latent attention&nbsp;&nbsp; which uses low rank latent representations&nbsp; to cut memory use and speed up attention.&nbsp;&nbsp; You will see that in models like the Youtu LLM,&nbsp; DeepSeek V3 and Mistral 4. And to close it out,&nbsp;&nbsp; we have a wild one. 8B plus parameter models&nbsp; in the browser are now real. Here's GPT-OSS,&nbsp;&nbsp; a 20B parameter model that runs at 40 tokens&nbsp; per second in the browser. The reason this is&nbsp;&nbsp; even possible is mixture of experts. Each&nbsp; token only activates a subset of expert&nbsp;&nbsp; networks instead of the entire model. That keeps&nbsp; inference fast enough to be practical powered by&nbsp;&nbsp; custom MoE and QMoE ops in ONNX runtime. It's&nbsp; also the same idea behind LFM2 MoE which shows&nbsp;&nbsp; how models this large can stay efficient enough&nbsp; [clears throat] for real ondevice inference. Beyond the new model architectures, we've added&nbsp; a brand new model registry and this one is a&nbsp;&nbsp; big deal for anyone building production apps.&nbsp; [snorts] ModelRegistry gives you full visibility&nbsp;&nbsp; and control over the required files. You can&nbsp; discover exactly which files a model needs,&nbsp;&nbsp; calculate the total download size, check whether&nbsp; the model is already cached, and also clear the&nbsp;&nbsp; cache when needed. Thanks to Steven Roussey, a&nbsp; very active member of our community, you can also&nbsp;&nbsp; check which precision types are available for&nbsp; any given model ID. Based on the ModelRegistry,&nbsp;&nbsp; we can now expose a new "progress_total"&nbsp; callback. It gives you overall loading&nbsp;&nbsp; progress while a model downloads. So you can show&nbsp; accurate loading bars without tracking individual&nbsp;&nbsp; files yourself. Two new environment settings&nbsp; give you more control over how Transformers.js&nbsp;&nbsp; loads models. First env.useWasmCache. WASM&nbsp; runtime files are now cached by default. So&nbsp;&nbsp; once a model is loaded, your app can work fully&nbsp; offline on later visits. You can still opt out,&nbsp;&nbsp; but it's enabled automatically&nbsp; whenever a cache is available.&nbsp;&nbsp; Second, env.fetch. You can now pass your own&nbsp; fetch function with custom headers, auth tokens,&nbsp;&nbsp; or abort signals. And Transformers.js will&nbsp; use it for all network requests. This is&nbsp;&nbsp; great for authenticated model access or for&nbsp; integrating with your own request pipeline. Last but not least, here's a quick rundown of the&nbsp; tooling improvements. The entire codebase has been&nbsp;&nbsp; restructured, refactoring the monolithic models&nbsp; file into separate per model files. Projectwide&nbsp;&nbsp; formatting with prettier migration to PNPM&nbsp; workspaces. TypeScript users get improved&nbsp;&nbsp; types with dynamic pipeline output types.&nbsp; So, TypeScript now knows the exact return&nbsp;&nbsp; type based on the task you're using. And a switch&nbsp; to esbuild. Build times dropped from two seconds&nbsp;&nbsp; to 200 milliseconds and file sizes are slightly&nbsp; smaller. The Tokenizer is now its own standalone&nbsp;&nbsp; package. So @huggingface/tokenizers is an 8.3&nbsp; kilobytes zero dependency library that works&nbsp;&nbsp; everywhere.
