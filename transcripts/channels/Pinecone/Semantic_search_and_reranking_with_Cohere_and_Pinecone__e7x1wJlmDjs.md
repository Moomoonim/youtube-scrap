PineCone이 이러한 기능을 수행할 수 있는 이유는 구현 방식 때문입니다. PineCone은 서버리스 방식으로 작동하여 컴퓨팅 및 스토리지 비용을 분리합니다. 즉, 쿼리 부하가 증가하면 PineCone이 확장되고, 업서트되는 데이터 양이 증가하면 PineCone이 자동으로 확장됩니다. 따라서 이러한 리소스를 개별적으로 프로비저닝할 필요가 없습니다. PineCone을 사용하여 이러한 아키텍처를 구현하면 다음과 같은 구조가 됩니다. 이것이 바로 시맨틱 검색 애플리케이션입니다. 데이터 소스에는 관련 메타데이터가 포함될 수 있으며, 쿼리를 임베딩으로 변환하는 임베딩 모델도 필요합니다. 특히 Vector 데이터베이스에 업서트할 때 이 임베딩 모델이 필요합니다. 변환된 쿼리는 PineCone 데이터베이스와 상호 작용하는 API로 전송됩니다. 쿼리를 전달하면 해당 결과가 반환되고, 이것이 바로 시맨틱 검색 결과입니다. 이러한 아키텍처의 어려운 점은 설정해야 할 구성 요소가 많다는 것입니다. 예를 들어 임베딩 모델을 호스팅하거나 배포해야 할 수 있고, 데이터 처리 방식을 설정해야 할 수도 있습니다. UPS가 시작되면 데이터가 어떻게 청크되는지 처리해야 하고, 물론 여러 위치나 엔티티에서 데이터를 업서트하거나 쿼리하는 경우 이 모든 것을 조정해야 합니다. 통합 추론이 출시되면서 이제 PineCone을 사용하여 임베딩 및 추상화를 수행하는 데 필요한 API 호출이 하나뿐입니다. 이전에는 하나의 API 호출로 데이터를 직접 임베딩하고 다른 API 호출 로 다시 업서트해야 했습니다. 엔드포인트에 텍스트를 전달하면 임베딩되어 PineCone에 업서트되고, 텍스트를 사용하여 직접 다시 쿼리할 수 있습니다. 이는 시맨틱 검색의 작업 부하를 크게 단순화합니다. 또한 Vector 데이터베이스가 있는 위치 근처의 인프라에 최첨단 배포 모델을 호스팅하고 있기 때문에 속도가 더 빠릅니다. 이미 매우 우수한 모델을 선정했으며, 오늘은 그중 하나인 재순위화(reranking) 모델에 대해 이야기해 보겠습니다. 인덱스를 관리할 때 할 일은 사용할 모델과 해당 인덱스가 생성되는 방식을 지정하는 것뿐입니다. 인덱스 생성 호출을 사용하면 인덱스를 생성할 뿐만 아니라 차원 크기를 지정하고 이후 요청에 맞게 설정하는 것이 매우 간편해집니다. 인덱스 생성 후 업서트 및 임베딩에는 네임스페이스와 업서트할 레코드를 지정하는 단 한 줄의 코드만 필요합니다. 검색 시에도 별도의 재순위 지정 호출이 필요하지 않습니다. 여기에서처럼 CoherentRank 3.5와 같은 재순위 지정 값을 입력 쿼리 텍스트 및 기타 하이퍼파라미터와 함께 지정하여 관심 있는 내용과 관련된 결과를 얻을 수 있습니다. 이는 매우 편리하며 최소 두세 번의 추가 API 호출을 방지합니다. 오늘 노트북에서 이 부분을 시연해 보겠습니다. 하지만 일반적인 시맨틱 검색을 구현할 때 문제가 발생합니다. 시맨틱 검색만으로는 충분하지 않은 사용 사례가 많습니다. 예를 들어 두 가지 특정 조건을 충족해야 하거나 초기 시맨틱 검색에서 반환된 결과를 더욱 세분화해야 하는 경우입니다. 일반적으로 관련성이 있는 것과 없는 것을 매우 구체적으로 정의해야 하거나, 하위 검토 프로세스로 보내야 하는 문서 수에 제한이 있는 도메인에서 작업할 때(이를 컨텍스트 최적화라고 합니다), 시맨틱 검색에서 반환되는 결과를 추가적으로 정리해야 파이프라인이 제대로 작동할 수 있습니다. 랭커는 시맨틱 검색에서 반환된 초기 문서 집합을 정제하여 결과를 최대한 관련성 있게 만드는 데 도움이 되는 또 다른 모델입니다. 관련성 측면에서 랭커는 사용자 쿼리와 문서를 입력받아 해당 쿼리와 반환 문서의 컨텍스트를 고려한 쌍별 점수를 계산합니다. 그런 다음 이러한 모든 점수를 취하여 가장 높은 점수부터 가장 낮은 점수 순으로 순위를 매기고 가장 높은 점수를 받은 문서의 부분 집합을 반환합니다. 즉, 하위 프로세스 에 더 관련성이 높은 문서의 수를 줄이는 것입니다. 랭커는 각 사용자 쿼리-문서 쌍에 대해 문서와 사용자 쿼리를 동시에 고려하기 때문에 점수는 컨텍스트에 맞춰 제공됩니다. 반환되는 문서는 주어진 워크플로에 가장 관련성이 높은 문서입니다. 둘째로, 의미 검색을 다른 더 큰 아키텍처 내에서 수행할 때 컨텍스트 최적화라는 개념이 있습니다. 이 경우 청크를 사람 검토자에게 전달하거나 대규모 언어 모델(LLM)에 전달해야 합니다. 오늘 다루지는 않지만, 이 작업을 컨텍스트 최적화 문제로 접근하는 것이 유용합니다. 컨텍스트가 너무 많으면 검토자나 LLM이 중요한 정보를 찾는 데 필요한 시간을 많이 소모할 수 있습니다. 반대로 컨텍스트가 너무 적으면 하위 작업을 완료하는 데 실제로 중요한 문서를 놓칠 수 있습니다. 하지만 랭커를 사용하면 컨텍스트를 최적화하여 반환되는 문서가 구현하려는 하위 프로세스에 가장 관련성이 높도록 할 수 있습니다.
